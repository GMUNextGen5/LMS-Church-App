1. Frontend Implementation: UI/UX for a 3-tier user system.
2. Access Control: A robust 3-Tier Access Control System (Admin, Teacher, Student/Parent).

4. AI Integration: Secure implementation of Gemini API features.
5. Data Analytics: A data-pipeline architecture for BI tools (Power BI, Excel).
6. Deployment: Hosting the application on a global CDN (Cloudflare).

1. Core System Architecture
1.1. Technology Stack & Rationale
This system uses a "serverless" stack, meaning we do not manage or scale our own backend servers, leading to lower cost and maintenance.
* Frontend: Static HTML, CSS (Tailwind), and JavaScript.
   * Rationale: Simple, fast, and universally compatible. Easy to host on any static CDN.
* Hosting: Cloudflare Pages.
   * Rationale: Provides a high-performance global CDN for free, ensuring low-latency access for users anywhere.
* Backend (BaaS - Backend-as-a-Service): Google Firebase
   * Authentication: Manages user identity (logins, sessions, passwords).
      * Rationale: Secure, fully-managed, and supports Email/Password, Google, etc., out-of-the-box.
   * Database: Cloud Firestore.
      * Rationale: A real-time NoSQL database. Its primary benefit is real-time data sync (updates appear instantly) and a powerful, server-side security model.
   * Security: Firestore Security Rules.
      * Rationale: This is the core of our security. It provides server-side enforcement of all data access policies, preventing any unauthorized data access, even if the client-side code is compromised.
   * Serverless Functions: Firebase Cloud Functions (Node.js).
      * Rationale: Used for secure, server-side actions like calling the Gemini API, hiding API keys, and performing complex automated tasks.
1.2. High-Level System Diagram
This diagram shows the "triangle" of a modern serverless app:
1. User: Interacts with the Frontend.
2. Frontend (Cloudflare): Serves the application UI.
3. Backend (Firebase): The UI (on the user's browser) communicates directly and securely with Firebase services.
1.3. Detailed Data & Security Flow
1. User visits your-lms.pages.dev. Cloudflare Pages serves the lms_prototype.html file.
2. The JavaScript in the HTML file initializes the Firebase connection.
3. The user is shown a Login form. They enter their credentials.
4. The JS client calls signInWithEmailAndPassword(auth, email, password). Firebase Authentication validates the credentials and returns a secure JSON Web Token (JWT) to the client. This token secretly contains their uid.
5. The client's Firebase SDK now automatically includes this JWT in all subsequent requests.
6. The user's role is fetched from the users/{uid} document in Firestore.
7. The user (now logged in as a "Parent") clicks to view their child's grades.
8. The JS client requests the data from students/{studentId}/grades.
9. Firestore Security Rules (on Google's server) intercept the request. The rules check:
   * a) "Does this request have a valid JWT?" (Yes)
   * b) "What is the uid from the token?" (e.g., parent-uid-123)
   * c) "Does parent-uid-123 have permission to read this specific document?"
   * d) The rule (allow read: if request.auth.uid == resource.data.parentUid;) is evaluated. It passes.
10. Firestore sends the data only for that student back to the client.
2. Implementation: Frontend & UI
2.1. UI Structure (Login, App)
Your HTML file will have two main "states," managed by JavaScript:
1. Logged Out State: A full-page container (e.g., <div id="auth-container">...</div>) that shows the Login and Sign-Up forms. The main app (<div id="app-container">...</div>) is hidden with style="display: none;".
2. Logged In State: When onAuthStateChanged reports a user is logged in, you hide #auth-container and show #app-container.
2.2. Role-Based View Logic
This is how you implement the 3-tier view in the client-side JavaScript.
1. Fetch Role on Login: After a user logs in, fetch their role from Firestore.
// Global variable
let currentUserRole = null;

onAuthStateChanged(auth, async (user) => {
   if (user) {
       // User is logged in
       const userDocRef = doc(db, "users", user.uid);
       const userDoc = await getDoc(userDocRef);

       if (userDoc.exists()) {
           currentUserRole = userDoc.data().role; // e.g., "teacher", "student"
       } else {
           // This is a new sign-up, role might not be set yet.
           // Or, an error.
           currentUserRole = "student"; // Default
       }

       // Now, show the app and configure the UI
       document.getElementById('auth-container').style.display = 'none';
       document.getElementById('app-container').style.display = 'block';
       configureUIForRole(currentUserRole);
   } else {
       // User is logged out
       currentUserRole = null;
       document.getElementById('auth-container').style.display = 'block';
       document.getElementById('app-container').style.display = 'none';
   }
});

2. Configure UI: A single function shows/hides elements based on the role.
function configureUIForRole(role) {
   // Hide all role-specific elements by default
   document.querySelectorAll('.admin-only, .teacher-only, .student-only').forEach(el => {
       el.style.display = 'none';
   });

   // Show elements for the specific role
   if (role === 'admin') {
       document.querySelectorAll('.admin-only, .teacher-only').forEach(el => {
           el.style.display = 'block'; // Admin sees teacher + admin
       });
   } else if (role === 'teacher') {
       document.querySelectorAll('.teacher-only').forEach(el => {
           el.style.display = 'block'; // Teacher sees only teacher
       });
   } else {
       document.querySelectorAll('.student-only').forEach(el => {
           el.style.display = 'block'; // Student/Parent sees their view
       });
   }
}

3.2. Authorization (AuthZ): Defining & Provisioning User Roles
This is the "AuthZ" part: what a user is allowed to do.
      1. Data Model: As shown in 3.1, a users/{uid} collection stores the role.
{ "email": "teacher@school.com", "role": "teacher" }

      2. Provisioning Admin/Teacher Roles: Since all new sign-ups default to "student", you need a secure way to "promote" users.
         * Method 1 (Manual): An Admin goes into the Firebase Console (the database view) and manually changes the role field for a new teacher's uid from "student" to "teacher". This is simple but not scalable.
         * Method 2 (Admin UI): Create an "Admin" panel in the app (only visible to role == 'admin'). This panel would list all users, and an Admin could change their role. This action must be secured by Firestore Security Rules (e.g., only an Admin can write to the role field).
         * Method 3 (Cloud Function): Create a secure "invite" system. An Admin enters a new teacher's email. A Cloud Function creates an "invite" document, and when the user signs up with that email, another function verifies the invite and assigns the "teacher" role automatically.
4. Implementation: Secure Data Model (FERPA)
4.1. Evolved Firestore Data Structure
         * users/{uid}: Stores role (see 3.2).
         * students/{studentId}: Main student profile.
         * Rationale: This document links the student to their login (studentUid) and their parent's login (parentUid). These can be the same uid if the parent uses one login for everything, or different uids. <!-- end list -->
{
 "name": "John Smith",
 "parentUid": "parent-uid-456",
 "studentUid": "student-uid-789"
}

         * students/{studentId}/attendance/{date}: Sub-collection.
         * Rationale: Using sub-collections is vital. It bypasses the 1MiB document size limit, allowing a student to have infinite attendance records.
         * students/{studentId}/grades/{gradeId}: Sub-collection.
         * Rationale: Same as above. Prevents a single student document from becoming too large.
         * courses/{courseId}: New collection.
         * Rationale: This is the "join" table that links teachers to students. This is critical for the security rules. <!-- end list -->
{
 "courseName": "Biology 101",
 "teacherId": "teacher-uid-abc",
 "studentIds": ["student-id-1", "student-id-2"] // List of student Doc IDs
}

4.2. Firestore Security Rules (The "Server-Side Enforcer")
This is the most important file in the project. It runs on Google's servers and cannot be bypassed. This is your primary FERPA compliance tool.
rules_version = '2';
service cloud.firestore {
 match /databases/{database}/documents {

   // --- Helper Functions ---
   // Get the requesting user's role from their user document
   function getUserRole() {
     return get(/databases/$(database)/documents/users/$(request.auth.uid)).data.role;
   }
   
   // Check if the user is an Admin
   function isAdmin() {
     return getUserRole() == 'admin';
   }
   
   // Check if the user is a Teacher
   function isTeacher() {
     return getUserRole() == 'teacher';
   }

   // --- User Profile Rules ---
   // A user can read/update their own profile.
   // An Admin can read/update anyone's profile.
   // An Admin is the only one who can change the 'role' field.
   match /users/{uid} {
     allow read: if request.auth.uid == uid || isAdmin();
     allow create: if request.auth.uid == uid;
     allow update: if (request.auth.uid == uid && !('role' in request.resource.data)) || isAdmin();
   }

   // --- Student Data Rules ---
   match /students/{studentId} {
   
     // Helper: Check if this user is the parent or the student
     function isOwner() {
       return request.auth.uid == resource.data.parentUid || 
              request.auth.uid == resource.data.studentUid;
     }
     
     // Helper: Check if this user is a teacher assigned to this student
     // This rule checks if there is any course doc where the requesting
     // user is the teacher AND this studentId is in that course's studentIds list.
     function isAssignedTeacher() {
       return isTeacher() && exists(
         /databases/$(database)/documents/courses/$(courseId)
         where get(/databases/$(database)/documents/courses/$(courseId)).data.teacherId == request.auth.uid
         && studentId in get(/databases/$(database)/documents/courses/$(courseId)).data.studentIds
       );
     }

     // ** THE FERPA-COMPLIANT RULE **
     // READ: Allowed by Admin, Owner (Parent/Student), or Assigned Teacher.
     allow read: if isAdmin() || isOwner() || isAssignedTeacher();
     
     // WRITE (Update/Delete): Allowed by Admin or Assigned Teacher ONLY.
     // This explicitly blocks Parents/Students from changing records.
     allow update, delete: if isAdmin() || isAssignedTeacher();
     
     // CREATE: Only an Admin can create new student profiles.
     allow create: if isAdmin();

     // --- Sub-Collection Rules (Grades & Attendance) ---
     // These rules automatically inherit the logic from the parent /students/{studentId}
     // We just need to define their own specific logic.
     
     match /grades/{gradeId} {
       // Read access is inherited from parent (Admin, Owner, Teacher)
       allow read: if isAdmin() || isOwner() || isAssignedTeacher();
       
       // Write access is ONLY for Admin or Teacher
       allow create, update, delete: if isAdmin() || isAssignedTeacher();
     }
     
     match /attendance/{date} {
       // Read access is inherited
       allow read: if isAdmin() || isOwner() || isAssignedTeacher();
       
       // Write access is ONLY for Admin or Teacher
       allow create, update, delete: if isAdmin() || isAssignedTeacher();
     }
   }
   
   // --- Course Rules ---
   // Admins can create/delete courses.
   // Only the assigned teacher or an Admin can update a course.
   match /courses/{courseId} {
     function isCourseTeacher() {
        return request.auth.uid == resource.data.teacherId;
     }
     allow read: if isAdmin() || isCourseTeacher();
     allow create: if isAdmin();
     allow update, delete: if isAdmin() || isCourseTeacher();
   }
 }
}

5. Implementation: Gemini API Integration
5.1. Feature Overview (Summaries, Study Tips)
As seen in the prototype, the app will call the Gemini API to:
         1. Generate performance summaries (using grades + attendance).
         2. Generate actionable study tips (using grades).
5.2. Crucial Security Note: Why Client-Side Calls Are Unsafe
The prototype calls the Gemini API directly from the browser. DO NOT DO THIS IN PRODUCTION. Even if the API key is "handled by the environment," it is still exposed to the client. This allows an attacker to:
         1. Steal your API key.
         2. Use it in their own app, running up your bill.
         3. Send abusive or unintended queries, getting your project banned.
5.3. Secure Architecture: Using Firebase Cloud Functions
The only secure way to call a private API is from a server. We will use a Firebase Cloud Function as a "proxy" or "middleman."
         1. Client (Browser): The user clicks "Generate Summary." The client does not call Gemini. Instead, it calls our own Cloud Function, passing only the studentId.
// This is a NEW function in your client-side JS

   const studentId = gradeSelect.value;
   const getSummaryFunction = httpsCallable(functions, 'getPerformanceSummary');
   const result = await getSummaryFunction({ studentId: studentId });
   showModalResult(result.data.summaryHtml);

         2. Server (Cloud Function): You will deploy this Node.js code to Firebase.
            * index.js (inside your functions folder): <!-- end list -->
const { onCall } = require("firebase-functions/v2/https");
const { getFirestore } = require("firebase-admin/firestore");
const { initializeApp } = require("firebase-admin/app");

// Initialize Firebase Admin SDK
initializeApp();

// Import the Google AI SDK
const { GoogleGenerativeAI } = require("@google/generative-ai");
// Get API key from secure storage
const genAI = new GoogleGenerativeAI(process.env.GEMINI_API_KEY);

exports.getPerformanceSummary = onCall(async (request) => {
   // 1. Verify user is authenticated
   if (!request.auth) {
       throw new functions.https.HttpsError("unauthenticated", "User must be logged in.");
   }

   const studentId = request.data.studentId;
   const uid = request.auth.uid;
   const db = getFirestore();

   // 2. Verify user is ALLOWED to get this summary
   // (This is re-checking your security rules on the server!)
   const studentRef = db.doc(`students/${studentId}`);
   const studentDoc = await studentRef.get();
   if (!studentDoc.exists()) throw new HttpsError("not-found", "Student not found.");

   const isOwner = uid === studentDoc.data().parentUid || uid === studentDoc.data().studentUid;
   // (Add teacher logic here too)

   if (!isOwner) { // && !isTeacher && !isAdmin) {
        throw new HttpsError("permission-denied", "You do not have access to this student.");
   }

   // 3. User is authenticated and authorized. Fetch the data.
   const gradesSnapshot = await studentRef.collection("grades").get();
   const grades = gradesSnapshot.docs.map(doc => doc.data());

   // 4. Call the Gemini API (from the server, securely)
   const model = genAI.getGenerativeModel({ model: "gemini-2.5-flash-preview-09-2025" });
   const systemPrompt = "You are a helpful teacher...";
   const userPrompt = `Generate summary for grades: ${JSON.stringify(grades)}`;

   const result = await model.generateContent({
       contents: [{ parts: [{ text: userPrompt }] }],
       systemInstruction: { parts: [{ text: systemPrompt }] },
   });

   const text = result.response.candidates[0].content.parts[0].text;

   // 5. Return the result to the client
   return { summaryHtml: text };
});

6. Implementation: Data & BI Integration
6.1. KPI Dashboards (Power BI via BigQuery)
            1. Firebase BigQuery Extension: In the Firebase Console, go to Extensions and install the "Export Collections to BigQuery" extension.
            2. Configure:
            * Point it to students.
            * Enable the "Sync subcollections" option.
            * This extension auto-creates a BigQuery dataset and "shadow" tables that mirror your Firestore data.
            3. Create SQL Views: The raw exported data is nested. In BigQuery, write a simple SQL View to flatten the data.
-- Example View: `v_student_grades`
SELECT
 studentId,
 JSON_VALUE(data, '$.name') as student_name,
 JSON_VALUE(data, '$.memberId') as member_id,
 grade.assignmentName,
 grade.score
FROM
 `your_project.students_raw_latest`
CROSS JOIN
 UNNEST(JSON_QUERY_ARRAY(data, '$.grades')) AS grade

            4. Connect Power BI:
               * In Power BI Desktop, click Get Data > Google BigQuery.
               * Sign in and select your BigQuery project.
               * Power BI will show your new SQL View (v_student_grades) as a clean, flat table ready for building dashboards.

7. Implementation: Deployment & Hosting
7.1. Hosting with Cloudflare Pages
                  1. Create GitHub Repo: Push your lms_prototype.html file to a new GitHub repository.
                  2. Create Cloudflare Project:
                  * Log in to Cloudflare and go to Workers & Pages.
                  * Click Create application > Pages > Connect to Git.
                  * Select your new GitHub repository.
                  * In Build settings, select "None" as the "Framework preset."
                  * Click Save and Deploy.
                  3. Your app will be live at your-project-name.pages.dev.
7.2. Critical: Firebase CORS & Domain Authorization
Firebase will block your live Cloudflare app from logging in. You must authorize the domain.
                  1. Go to the Firebase Console.
                  2. Select your project, then go to Authentication.
                  3. Click the Settings tab.
                  4. Go to the Authorized domains section.
                  5. Click Add domain.
                  6. Enter your Cloudflare Pages domain (e.g., your-project-name.pages.dev).
                  7. Click Add.
This tells Firebase that it is safe to accept login requests from your hosted application.
8. Conclusion
This architecture (Static Frontend + Firebase BaaS) provides a complete, professional-grade solution. It separates concerns cleanly:
                  * Cloudflare handles global, fast delivery of the UI.
                  * Firebase Authentication handles identity.
                  * Firestore handles real-time data.
                  * Firestore Security Rules provide a server-side "firewall" for all data, ensuring compliance and security.
                  * Cloud Functions provide a secure server environment for sensitive operations like calling AI models.
                  * BigQuery Extensions provide an automated data pipeline for analytics.
This model is infinitely scalable, secure by default, and has a low maintenance burden, allowing you to focus on building features rather than managing infrastructure.
